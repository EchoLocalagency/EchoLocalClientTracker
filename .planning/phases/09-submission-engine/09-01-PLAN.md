<task type="auto">
  <name>Task 2: Build submission engine with rate limiter, state machine, NAP audit, and screenshot capture</name>
  <files>scripts/seo_engine/submission_engine.py</files>
  <action>
Create `scripts/seo_engine/submission_engine.py` following the discovery.py CLI pattern and the research skeleton from 09-RESEARCH.md. The script must implement all five SUB requirements:

**CLI interface** (follows discovery.py pattern):
- `--client-slug` (required): Client slug e.g. mr-green-turf-clean
- `--dry-run`: Show eligible directories and rate limit status without submitting
- `--limit N` (default 5): Max submissions this run (still capped by daily/weekly limits)
- Module invocation: `python3 -m scripts.seo_engine.submission_engine --client-slug X`

**Main flow** (`run_submission_engine(client_slug, dry_run, run_limit)`):
1. Load client from `clients` table by slug, then load `client_profiles` by client_id
2. Check rate limits (SUB-02): query submissions table for today's and this week's submitted count. If either cap reached, print message and exit
3. Query eligible directories: `tier=3 AND captcha_status='no_captcha' AND enabled=true`. Filter by client trade tags (match directory trades array against client trade, or empty trades = universal directory)
4. Query existing submissions for this client to skip directories already submitted/verified/skipped
5. For each eligible directory (up to run_limit and remaining daily/weekly budget):
   a. NAP audit (SUB-04): build form_data dict from profile + form_config, compare against canonical profile. If mismatches, mark submission `skipped` with mismatch details in notes, continue to next
   b. Open browser with stealth (SUB-01): use `Stealth().use_async(async_playwright())`, create context with realistic viewport (1280x900), user-agent (Chrome 120 on macOS), locale en-US
   c. Navigate to submission_url, wait for networkidle
   d. Runtime CAPTCHA check: `page.evaluate("() => !!(window.grecaptcha || window.hcaptcha || document.querySelector('[data-sitekey]'))")`. If detected, update directory captcha_status, mark submission skipped, continue
   e. State machine (SUB-03): advance through form_loaded -> form_filled -> post_sent stages, persisting each to submissions.metadata jsonb (read-then-merge pattern to preserve existing metadata keys)
   f. Fill form fields using form_config locator strategies with human-like typing: `press_sequentially(text, delay=random.randint(80, 180))` + `page.wait_for_timeout(random.randint(200, 600))` between fields. Try each locator strategy in priority order (label first, placeholder second, css third). Skip fields where no locator matches.
   g. Click submit button: try `get_by_role("button", name="Submit")`, fallback to `get_by_role("button", name="Send")`, fallback to `locator("button[type='submit']")`, fallback to `locator("input[type='submit']")`
   h. Wait for confirmation: `page.wait_for_url()` with 10s timeout OR success text pattern. Log outcome.
   i. Mark submission `submitted` with `submitted_at` timestamp

**Error handling** (SUB-03 + SUB-05):
- In except block: read current stage from submissions.metadata
- If stage is `post_sent`: mark status `submitted` (not failed) with note about post-submit error
- If stage is before `post_sent`: capture full-page screenshot to `screenshots/{client_slug}/{dir_domain}_{timestamp}.png`, store screenshot path + error + truncated traceback in metadata, mark status `failed`
- Never retry failed submissions automatically

**Rate limiter** (SUB-02):
- `check_rate_limits(sb, client_id)` returns (today_count, week_count) or raises RateLimitExceeded
- Check before each individual submission, not just at start
- Daily cap: 5, Weekly cap: 8 (constants at module level)

**NAP audit** (SUB-04):
- `nap_audit(profile, form_data)` returns list of mismatch strings
- Compares: business_name, phone, address_city, address_state, website
- Any mismatch -> skip submission with details in notes

**Important implementation notes:**
- Use `await page.wait_for_timeout(ms)` not `time.sleep()` (async-safe)
- One browser per submission run, one page per directory (close and reopen page between directories)
- Check for Playwright browser binary at startup, print clear error if missing
- Log each step to stdout with emoji-free, clear status messages
- Use read-then-merge for metadata updates: `existing = sub["metadata"] or {}; existing.update(new); sb.update({"metadata": existing})`
- The script is designed for sequential single-client runs. Add a note in the docstring about this.
